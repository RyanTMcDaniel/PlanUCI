SUPABASE SETUP : Set up 13 different tables for course information. I am pulling the info from AnteaterAPI. I am seperating entities based on how often the information changes (think Course Description vs Course Building Number). Doing this allows for safer updating as their is less info to overwrite on each 'refresh'. Mirrored my tables by the sshape of the data returned by API JSON responses to make storing easier. Setup RLS in Supabase to block unauthorized access. This keeps user-specific data safe and only readable by allowed users. Backend import scripts use the service-key role which bypasses RLS, so it never goes in frontent code or the public repo, only in the server environment variables.

Pulled UCI Catalog : Decided to pull everything rather than just STEM and GEs. Filtering would have led to gaps in courses and not been as extensive as I want. Current live counts (query Supabase directly; do not trust a number typed here): 7,771 courses, 3,281 instructors, 4,781 prereq_edges rows, 158 distinct major_id values across 87 distinct major_name programs, 55,880 grade_distributions rows, 1,481 rmp_reviews rows (was 3,264 before the matching fix below — the reduction is deliberate). Still haven't fully aligned prereqs and major requirements : I initially hoped to pull major requirements from AnteaterAPI but they aren't stored there, so I scrape each major's requirement page. STEM+Business majors are the most complete; humanities are thinner. Sections that read 'pick X of Y courses' don't scrape cleanly, so those get entered manually.

Difficulty Scoring NLP Model : Fine tuned all-MiniLM-L6-v2 for 3-class classification. Classification boundaries set using percentile splits (20/60/20) computed on the TRAINING SET ONLY and then applied unchanged to val and test; the split asserts zero course_id overlap (ml/data/preprocess.py). Fixed boundaries initially produced class imbalance with only 10/1674 courses labelled 'Hard' — GPA clusters too tightly for a uniform 1-10 scale to work. Started with a regression head trained on MSE; it collapsed, predicting 'medium' for essentially everything (macro F1 0.248, easy F1 and hard F1 both exactly 0.0 — see ml/evals/difficulty_nlp_v1_metrics.json). Switched to classification with cross-entropy and oversampled the easy/hard tiers, since ~60% of the data was labelled medium. Filtered out grad courses and participation-graded courses (assumed to be those with exactly 4.0 GPA average).

Test macro F1: 0.576, accuracy 62.7%, per tier F1 (easy 0.585, medium 0.697, hard 0.447). Produced by ml/evals/eval_difficulty.py → ml/evals/results/nlp_eval.json. NOTE: an earlier version of this doc reported 0.619 / 65.9%. That number was measured by an eval script that formatted its input as "Department: {dept}. Course: {id}. {description}" while the model was trained and is served on "{title}: {description}" — i.e. it scored the model on a format it had never seen. The eval now uses the training/serving format and 0.576 is the corrected figure. The redundant second eval script that produced the old number has been deleted.

Hard F1 is the weakest tier and always will be from text alone: a course description cannot tell you a professor curves brutally. That is the entire reason for blending in GPA and RMP signals downstream.

Difficulty Scoring Heuristic (NOT an "ensemble", NOT a trained model) : Combines three signals (NLP tier score, GPA average, RateMyProfessor difficulty) into a 1-10 difficulty score at two granularities: per-(course, professor) and course-level (sections-weighted rollup, which powers the schedule optimizer). Grade distributions store raw instructor strings that must be linked to instructor records: exact match via normalized shortened_name, multi-instructor co-taught sections with GPA attributed evenly across matched instructors, and a rapidfuzz fallback at threshold 95. Weights: NLP 0.358 / GPA 0.291 / RMP 0.465, produced by ml/models/fit_signal_weights.py → ml/models/signal_weights.json.

I originally called this a "Ridge Regression ensemble" and reported accuracy for it. That framing was wrong and I have removed it. Here is why, because the mistake is instructive:

The target the Ridge was fit against — `difficulty_score` — is itself constructed in ml/data/build_features.py as a renormalized weighted mean of the same three features, using the same weights the Ridge produces. The target is therefore a deterministic linear function of the model's own inputs: a closed-form weighted mean reconstructs it with MAE ~2e-8 and r = 1.0000000000. There is nothing to learn and nothing to measure. Every error metric I previously reported for it (a "5-fold CV MAE of 0.41", an "MAE of 0.855", a "42% improvement over NLP alone" and "67% over RMP alone") was computed against that circular target. None of those numbers are reproducible from anything in this repo and they have all been deleted. Worse, build_features.py READS signal_weights.json to build the target and fit_signal_weights.py READS the target to write signal_weights.json — a genuine chicken-and-egg loop, so the current weights are a fixed point of one round of self-training, not an estimate.

What the three weights honestly are: a PRIOR over how much each signal should count. They are stable and they are reasonable, and the Ridge-over-OLS reasoning still holds on its own terms — the three signals are correlated (a professor with harsh RMP ratings usually posts a lower GPA), OLS coefficients swing hard under resampling with correlated features, and L2 damps that. But a defensible weighting is not the same as a validated model, and I should not have claimed accuracy I had not measured.

To make this a real model I need an EXOGENOUS label — one not derived from the three inputs. The two candidates: (a) student-reported perceived difficulty collected via survey, which is the signal the product actually cares about and which none of the three proxies capture; or (b) next-term course GPA held out in time, predicting forward from prior terms, which at least makes the task a genuine forecast rather than a reconstruction. Until one of those exists, the blend ships as a documented heuristic and no accuracy figure is attached to it.

FastAPI Serving Layer : Chose FastAPI for ML serving because of native async support, automatic request validation, and auto-generated /docs endpoint which made endpoint testing faster. Models are loaded once at server startup from registry.json and kept in memory for lifetime of the server - trades higher memory usage for fast inference. Three endpoints return : course-level difficulty score, per-prof score with individual signal breakfown, and pro sentiment labels and confidence level.

Eval Framework : Standalone eval scripts, independent of training, producing held-out metrics. There is no perfect ground truth for difficulty — perceived difficulty and grade outcomes genuinely disagree, and a course with high grades can still demand enormous work. That is a real problem with the domain, but it is NOT a licence to report a metric against a target derived from the model's own inputs, which is what I was doing. The ensemble section of the eval has been deleted outright rather than caveated (see the heuristic entry above).

What survives, and it is the only ML accuracy claim this project makes: the difficulty tier classifier's held-out test macro F1, produced by ml/evals/eval_difficulty.py.

Sentiment Model Leakage : Retrained 2026-07-13. Test macro F1 dropped from 0.809 to 0.522 once a real leak was closed, and 0.522 is the honest number. Cause: instructor-to-RateMyProfessor name matching maps many distinct ucinetids onto the SAME RMP record, so the identical review blob — and the identical ratings, hence the identical weak label — was duplicated across up to 80 rows. Of 1,291 labelled rows only 591 texts were distinct. A stratified split over raw rows put 56% of the test set's exact review_text into train, so the model was scoring by memorizing strings it had already seen. train_sentiment.py now de-duplicates on review_text before splitting and asserts that no review_text appears in more than one split, failing loudly if it does. Metrics are written by the training run itself (ml/models/train_sentiment.py → sentiment_v1/model_card.json), so the published number can no longer drift from the checkpoint that produced it.

The underlying data bug has since been fixed — see "RMP Instructor Matching" below. match_professors.py has been DELETED: it fuzzy-matched at WRatio >= 88 and reassigned rmp_reviews.ucinetid, so running it would have silently undone the verified 1:1 mapping and reintroduced the collision. Nothing called it.

RMP Instructor Matching (2026-07-13) : Fixed the largest defect in the data. The old fetch_rmp.py had NO name comparison at all — it fired an instructor's name at RateMyProfessor's relevance search, regex-scraped every legacyId off the results page, took the first three, and kept whichever had the MOST RATINGS. RMP's search matches on any token including a bare middle initial, so "Michael A Carroll" returns the professor literally named 'A' Pantano (Mathematics, 58 ratings), who wins on popularity. Every UCI instructor with middle initial "A" got her ratings — 80 of them. The RMP-side firstName/lastName/department were being fetched by the GraphQL query and then thrown away, so the verification key was in hand and discarded. Measured damage: 3,231 instructors with an RMP rating drew from only 1,487 distinct RMP records; 2,276 (70.4%) shared a rating with someone else; 18,483 of 26,912 feature rows (68.7%) had a poisoned rmp_score, touching 4,964 of 5,769 courses.

The matcher now requires exact first-AND-last name agreement after normalization (middle initials/suffixes stripped, RMP records with a bare-initial first name rejected), uses department to disambiguate genuine namesakes (there really are two Michael Greens), and enforces one RMP record → at most one ucinetid, dropping BOTH claimants when contested rather than guessing. Every drop is logged with a reason (backend/scripts/rmp_dropped.log, 1,800 entries). Result: 1,481 verified 1:1 matches (45.1%), zero collisions. Coverage fell from 98% to 45% and that is the correct trade — a missing signal has a calibrated fallback, a wrong one silently poisons every difficulty score in the product. The invariant is now enforced by a Postgres partial unique index on rmp_id, not by a Python assertion that only fires if someone remembers to run the script; rmp_id / rmp_first_name / rmp_last_name / rmp_department are persisted so any match is auditable without re-scraping.

Signal Standardization + Missingness Calibration (2026-07-13) : Two bugs surfaced while auditing the fallback path that the RMP fix was about to make load-bearing.

(1) The three signals were never on a common scale. nlp_score and rmp_score both centre near 5.6 on the nominal 1-10 range, but gpa_score = (4 - gpa)/3 * 10 is crushed into the bottom (mean 1.9) because UCI GPAs cluster near 3.4. Weights are only meaningful across comparable scales, so the frozen prior was in large part being overridden by a scale artifact. Each signal is now rank-normalized to 1-10 before the weights are applied.

(2) The missing-signal path was biased. It renormalizes the surviving weights, which only preserves scale if the dropped signal shares a distribution with the ones that remain. It didn't: dropping rmp (mean 5.57) and reweighting toward gpa (mean 1.9) pushed scores DOWN 0.674 on a 1-10 scale, with 65.7% of rows moving more than 0.5 — so a course whose professors happened to lack an RMP match scored systematically EASIER. Standardization alone halved this but left +0.349, because each signal is rank-normalized over the rows where IT is present and those populations differ. Final fix: on rows carrying all three signals, measure what each missingness pattern scores versus the full blend and subtract that offset (nlp+rmp -0.1013, nlp+gpa -0.3492, nlp-only -0.8659). Residual shift is now 0.0000 for every pattern, printed on every build. The blend weights were NOT re-fit; signal_weights.json stays frozen. This fixed the scale, not the prior.

Also fixed: avgDifficulty == 0 on RMP means "no ratings on record", not "trivially easy". The old code fed it through the linear rescale and produced rmp_score = -1.25, off the bottom of the scale. EECS160B was scored 1.00/10 (easiest in the catalogue) almost entirely because of this. It is now treated as missing.

Blast radius of both fixes: mean absolute change in served course difficulty = 1.799 on the 1-10 scale; 66.4% of courses moved more than 1 point; 18.2% moved more than 3.

Per-Score Confidence (2026-07-13) : The missingness calibration makes a low-data score unbiased, but unbiased is not precise. After the RMP precision fix, 8,676 of 27,049 feature rows (32%) are scored by the NLP classifier alone — a text model with held-out macro F1 0.576 — and nothing in the product distinguished those from a score corroborated by three independent signals. A guess looked exactly as authoritative as a measurement.

Every score now carries a confidence tier keyed DIRECTLY on which signals backed it, not on any probabilistic estimate (there is no calibrated basis for one): high = nlp+gpa+rmp, medium = any two, low = nlp text alone. signals_present is emitted alongside so a score is auditable back to its inputs. As served, per course: high 2,319 (29.8%), medium 2,797 (36.0%), low 2,655 (34.2%). Threaded through /api/course-difficulties and rendered on the course card as FULL DATA / PARTIAL DATA / LOW DATA with the reason on hover. Low-confidence scores are shown, never hidden — the number plus an honest caveat beats withholding it. difficulty_score itself is unchanged and signal_weights.json stays frozen.
